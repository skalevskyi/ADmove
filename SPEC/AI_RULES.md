# AI Rules — SPM Project

Rules for AI-assisted development in Cursor and other tools for the Skalevskyi publicite mobile (SPM) project.

## Mandatory References

**Order of truth:** (1) **Behavioral truth** — **code / runtime** wins for **pricing logic, calculator behavior, API contracts, and what the app actually does**. (2) **Product visibility truth** — **corridor ranges + i18n** define all **user-facing** statements about audience / contacts; internal constants in code must **not** be surfaced as canonical reach numbers. (3) **`/SPEC/CALCULATOR_CURRENT_STATE.md`** — canonical runtime-facing behavior for the calculator **and** contact→lead structured fields / prefill (§6.2, §13). (4) **`/SPEC/PROJECT_CONTEXT.md`**, **`/SPEC/ROADMAP.md`**, **`/SPEC/DESIGN_SYSTEM.md`**. (5) **`/SPEC/AUDIENCE_PRICING_METHOD.md`** — methodology only; non-normative for runtime.

1. **Always check core SPEC before coding.** Read `PROJECT_CONTEXT.md`, `ROADMAP.md`, and when relevant `PRODUCT/website_spec.md`, `DESIGN_SYSTEM.md`. For **any calculator, lead payload, or contact prefill work**, read **`CALCULATOR_CURRENT_STATE.md`** first.
2. **Respect the architecture** defined in `/SPEC/PRODUCT/website_spec.md` for front-end and page structure where it matches `src/app/page.tsx`; if PRODUCT docs lag the repo, **code wins** — propose SPEC sync, do not regress runtime.
3. **Do not invent features** outside the roadmap. Check `/SPEC/ROADMAP.md` for pending scope (FAQ, map, media kit).
4. **Follow the product logic** in `/SPEC/PRODUCT/product_spec.md` for wording and value proposition when consistent with code.
5. **Keep implementation consistent** with PRODUCT pricing/visibility docs **as non-binding background**; **pricing** follows **`config.ts`** / engine; **user-facing contact ranges** follow **i18n corridor** per **`CALCULATOR_CURRENT_STATE.md`** §6 — not raw `INDICATIVE_MONTHLY_CONTACTS` in UI or messaging.

## Current phase & architecture (canonical)

- **Phase 2A (UI / Visual MVP)** is **complete**. **Phase 2B (functionality)** is **substantially complete** for shipped features (see `ROADMAP.md`).
- **Deployment:** Production is hosted on **Vercel** (Next.js **runtime**, App Router). Treat **Vercel** as the current canonical hosting model — **not** static-export-only or GitHub Pages–only. Follow existing `next.config` and deployment settings in-repo.
- **Lead API:** `POST /api/lead` — validation → **`persistLeadBackup`** (Upstash Redis, `persistence.ts`) → **`deliverLeadEmails`** (`provider.ts`: owner notification, then auto-reply). **`backup_failed`** → **503**. Backup OK + owner email fails → **200** + logs (`owner_email_failed_after_backup`). **Preserve** rate limit, honeypot, contracts; **`LEAD_BACKUP_SKIP`** dev-only.
- **Contact section:** **Functional** lead capture — submit, loading / success / error states, inline validation, premium success UX. **Do not** regress to a visual-only stub. The form submits **`packageId: null`** (optional field not wired from UI). **Structured payload:** **`leadOrigin`** (`'contact' \| 'calculator'`) and optional **`calculatorSummary`** when a calculator prefill is still associated at submit; owner email templates can render summary blocks. **Calculator prefill:** overwrite rules per **`CALCULATOR_CURRENT_STATE.md` §6.2 and §13** (refs cleared on manual edit; do not break `leadOrigin` / `calculatorSummary` semantics).
- **Lead emails — auto-reply:** Auto-reply is **implemented in code**. **External delivery** to arbitrary recipient addresses may require a **verified sender domain / sender identity** in Resend (infrastructure). That limitation is **not** treated as a “missing UI feature”; document as *implemented in code, pending verified sender domain for real external delivery* where relevant.
- **Still pending (roadmap):** FAQ, coverage map, media kit download — do not add without roadmap/SPEC update.

**Calculator:** Implemented (engine + UI). **Pricing** numbers and rules live in **`config.ts`** / **`rules.ts`** / **`engine.ts`**; **UI displays engine output for €** and must not become an alternate source of truth for pricing math. First-month discount on base media is **package-specific** (`FIRST_MONTH_DISCOUNT_EUR_BY_PACKAGE` in `config.ts` — see `/SPEC/CALCULATOR_CURRENT_STATE.md`). **Always refer to `/SPEC/CALCULATOR_CURRENT_STATE.md`** before changing calculator display or copy; **do not modify engine logic** (`engine.ts`, `rules.ts`) unless explicitly requested — prefer UI-only changes. **Indicative contacts (audience):** must be expressed using **corridor range strings** from i18n + separate unit line; **raw `INDICATIVE_MONTHLY_CONTACTS` values must not be used in UI or messaging** as headline reach figures. The **CPM** line is **static copy** (corridor-aligned), not engine-computed. The emerald **“included”** strip must **not** list `priority_booking` as a row (filter in `OfferCalculatorPanel`); config/engine inclusion rules stay unchanged.

**Landing structure remains fixed:** Navbar → **`<main>`** → Hero → **Process / Fonctionnement / Як працює** (section id `#support`) → **Parcours** (section id `#parcours`, implemented as **`VehicleSection.tsx`**) → Offres → Contact → **Footer** (**inside** `<main>` in `src/app/page.tsx`) → **`</main>`** → MobileBottomNav → FloatingActions. The Process section uses a concise 4-step explanatory flow (daily route, real exposure, repetition, accumulation) and does **not** include route breakdown or package/price numbers. Do **not** reintroduce separate top-level sections such as Problem, Solution, or an extra **Vehicle** slice beyond the existing Parcours block, nor Pricing, Coverage Map, Price Simulator, FAQ, or Media Kit unless the SPEC is explicitly updated. Visibility content lives inside Parcours; **fixed price tables are not shown** in Offres — the expandable panel uses the live calculator. **Format naming:** same identifiers appear in the Parcours visibility block with localized placement in parentheses (e.g. FR: arrière, latéral, habillage complet).

- **Hero (current):** DOM order is **not** “both CTAs then carousel”: implementation uses **primary CTA** after **scale**, then supporting copy + trust, **carousel**, then **secondary CTA** — see `HeroSection.tsx` and `PROJECT_CONTEXT.md`. i18n uses **`scaleSupporting`**, not a separate **`route`** key.

- **CTA shape consistency:** Prefer reusing **`ctaShapeBase`** from `src/lib/cta-shape.ts` for new full-width marketing CTAs unless a task explicitly requires a different pattern.

- **PWA / install:** Web manifest + `viewport.themeColor` + `globals.css` / `layout.tsx` theme boot behavior are part of the current launch contract; do not regress without checking `PROJECT_CONTEXT.md`.

- **Offres section:** Displays three packages (**BASIC**, **PRO**, **EXCLUSIVE** per i18n) with real mockups (**2 / 2 / 3** images per package in `OffresSection.tsx`; preview = first image; lightbox = marker navigation + swipe/keyboard). Do **not** display fixed price labels or a static pricing table in Offres. The per-card button (`calculer` in i18n) opens the shared **estimation panel** backed by the **real calculator** (selected format, duration, add-ons, summary, disclaimer, CTA to **#contact**). See `/SPEC/CALCULATOR_CURRENT_STATE.md`.
- **Parcours section:** Must remain a schematic route credibility block (timeline, route points, line). **Left column — live ping-pong (aller / retour):** **Aller:** segments fill **top → bottom** as the active stop advances. **Retour:** the return leg reads as movement on the **same** path: the route appears **established** (strong line toward Montpellier), then segments **behind** the active stop (already passed on the return) **mute** — do **not** regress to “a new path filling from the bottom.” **Markers on retour:** all non-active dots start **filled**; stops **passed** on the return trip **mute** progressively; upcoming stays at normal fill. **Aller markers:** past = filled, future = light — unchanged semantics. **Mobile layout:** Route module is a compact centered block on mobile (`w-fit` inner container, `md:w-full md:max-w-xs` on desktop); marker column left, labels right (single-row layout); do not center text under markers. **Right column:** **Semi-static** — the **location title** and **short description** are dynamic per active route point, with a compact **icon/tag** row under the title. The **three bullets** (Car / Repeat / MapPin) are **semi-dynamic** and change with the active route point; the previously repeated small note under the bullet list is **not** shown in the current UI. A subtle timeline hint under the left route module explains that clicking a route point pauses viewing and reveals its visibility context. Parcours content must focus on advertising exposure; avoid tourism or architecture language. Do not turn it into a map or add map tiles, map APIs, or map providers unless the roadmap phase allows it. **Visibility block** (inside Parcours): title from i18n `visibilityBlockTitle`; **numeric range** on one line, **unit** (e.g. `contacts / mois`) on the next — **no** “Range / Fourchette / Діапазон” prefix row; **BASIC / PRO / EXCLUSIVE** with localized placement in parentheses.
- **Public assets & optional base path:** If the app is served under a subpath, set `NEXT_PUBLIC_BASE_PATH` and use `BASE_PATH` / `withBasePath` from `src/lib/base-path.ts` for public URLs. Root Vercel deploys typically use an empty base path.
- **Mobile bottom nav and floating actions:** Are part of the current UI architecture. **Floating actions include only scroll-to-top** (no floating contact button). On mobile, bottom nav is the primary navigation; scroll-to-top appears above it after scroll threshold. **MobileBottomNav** must use a **mounted guard** to avoid hydration mismatch: initial render uses neutral state (no active item); active-section highlighting (hash + scroll) runs only after client mount via `useEffect`. Nav surface: **bg-white/80** and **dark:bg-slate-900/95** (match `MobileBottomNav.tsx`); blur and rounded shape preserved. Active switching must feel **discrete/mechanical** (press/release), with no shared sliding pill transfer between items and no geometry morphing between active/inactive states.
- **Reduced motion:** Support must be preserved. Respect `prefers-reduced-motion`; disable or simplify non-essential animations when set (hero rotator, Parcours autoplay, section entrances).
- **Motion list identity:** In Framer Motion animated lists, `key` must be locale-invariant. Do not use locale-visible strings (e.g. step titles) as `key`; use stable indices/ids so locale switching cannot cause partial/hidden animations.
- **UI layers:** When editing presentation, prioritize layout, styling, responsiveness, and animations. Use Tailwind and Framer Motion for subtle, professional motion. **Lead stack:** Preserve backup-before-email order, **`persistLeadBackup`**, **`deliverLeadEmails`** semantics, **error code shape** (incl. `backup_failed`), rate limit / honeypot, and auto-reply best-effort. **Analytics:** `trackSpmEvent` — do not remove conversion events without a deliberate product decision.
- **Imagery:** Prefer real vehicle imagery in Hero and Offres when assets exist. Route/coverage in Parcours must stay schematic; no real map integration until roadmap allows it.
- **Localization:** **FR-first, section-based architecture.** Translations live under `src/i18n/` with locales `fr`, `en`, `ua`; each locale is split into section files (common, hero, support, parcours, offres, contact). **FR is the canonical source;** EN and UA must match FR structure exactly. **Runtime:** custom LanguageContext (no next-intl/react-i18next); whole-locale fallback only (no key-by-key fallback). TranslationKeys are derived from FR. Proper nouns (place names) must use curated translations per locale; see DESIGN_SYSTEM.md. Do not rely on automatic translation for Montpellier, Port Marianne, Carnon, Palavas, La Grande-Motte (and UA equivalents: Монпельє, Порт-Маріанн, Карнон, Палавас, Ла-Гранд-Мотт). **Font loading:** the primary Manrope font loader must include the Cyrillic subset for UA to prevent font fallback and layout metric drift.

## Behaviour

- **Code/runtime is the source of truth** for **behavior** (what the app does: pricing, APIs, lead flow). SPEC documents must track code; if SPEC and code disagree on **behavior**, **fix SPEC** (not code) unless the task is explicitly to change product behavior. **Visibility messaging** is governed by **corridor + i18n**; do not document internal contact constants as user-facing truth.
- Use `PROJECT_CONTEXT` / `ROADMAP` for product scope and status; use **`CALCULATOR_CURRENT_STATE.md`** for calculator + lead-field behavior before changing `OfferCalculatorPanel`, `ContactSection`, or `/api/lead`.
- New features or copy changes should align with PRODUCT docs when possible; if PRODUCT lags the repo, **update SPEC** to match implementation.

## Documentation Synchronization Rule

- **Code** (engine, config, UI, API, lead layer) is always the source of truth for **implementation behavior**.
- **Product-facing visibility** (contact ranges, hero copy) must align with **corridor + i18n**; if internal `config` constants differ, they remain **technical** — document them as such, not as messaging truth.
- Documentation must reflect actual implementation; if mismatch occurs → **update documentation**, not code (unless explicitly changing product behavior).
- Never modify calculator **pricing** logic based on documentation assumptions alone.

**Verify before documenting or changing behavior:**

- Calculator: `OfferCalculatorPanel.tsx`, `engine.ts`, `rules.ts`, `config.ts`
- Contact → lead: `ContactSection.tsx`, `src/app/api/lead/route.ts`, `src/lib/lead/*` (incl. `persistence.ts`, `provider.ts`)

**`/SPEC/CALCULATOR_CURRENT_STATE.md`** is the canonical runtime-facing reference for **calculator behavior** and **§13 contact/lead structured fields**; keep it aligned when implementation changes.
